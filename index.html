Fundamentos e Conceitos

O que é UML?

UML (Unified Modeling Language), ou Linguagem de Modelagem Unificada, é uma linguagem padrão usada para modelar sistemas de software de forma visual. Ela permite criar representações gráficas (modelos) de um sistema, facilitando a visualização, a especificação, a construção e a documentação dos artefatos do software.

A UML oferece uma notação padronizada para expressar:

  • Estrutura estática de um sistema (classes, objetos)
  • Comportamento (interações, transições de estado)
  • Arquitetura (componentes, implantação)
  • Regras de negócio e requisitos (casos de uso)

História e Evolução

A UML surgiu nos anos 1990 como resultado da unificação de métodos de modelagem orientados a objetos. Seus criadores (conhecidos como os "Três Amigos": Grady Booch, Ivar Jacobson e James Rumbaugh) consolidaram suas metodologias em uma linguagem comum, posteriormente padronizada pelo OMG (Object Management Group).

1994-1995

Unificação inicial

Booch e Rumbaugh começam a unificar seus métodos (Booch'93 e OMT)

1995

OOSE se junta

Jacobson integra seu método OOSE no processo de unificação

1997

UML 1.0

Primeira versão adotada pelo OMG

2005

UML 2.0

Maior revisão com novos diagramas e conceitos

Atual

UML 2.5+

Refinamentos contínuos e ajustes de consistência

UML vs. Metodologia

É importante notar que a UML não é um processo ou metodologia de desenvolvimento em si. Ela não dita como conduzir um projeto ou a ordem de atividades, mas fornece uma notação independente de processo que pode ser aplicada em conjunto com qualquer metodologia (por exemplo, RUP, Scrum, etc.).

UML é UML não é
Uma linguagem visual Uma metodologia de desenvolvimento
Um conjunto de notações e semântica Um processo prescritivo
Uma ferramenta de comunicação Uma solução para todos os problemas
Um padrão aberto Limitado a orientação a objetos

Desenvolvedores usam a UML para visualizar e comunicar o design do sistema, identificar classes e componentes-chave e entender como as partes do sistema interagem.

Modelos vs. Diagramas

Um modelo UML é a representação abstrata completa de um sistema, podendo incluir vários diagramas diferentes que, juntos, descrevem múltiplos aspectos (estrutura estática, casos de uso, interações dinâmicas, etc.).

Já um diagrama UML é uma visualização específica, uma projeção do modelo que mostra certos elementos e relacionamentos sob uma certa perspectiva.

Analogia útil

Pense em um modelo UML como a planta completa de um edifício. Cada diagrama seria como uma vista diferente: planta baixa, vista frontal, instalações elétricas, etc. Todas são visualizações do mesmo edifício, mas destacando aspectos específicos.

Diagramas da UML

A UML fornece diversos tipos de diagramas, divididos em duas grandes categorias: diagramas estruturais (que mostram a estrutura estática do sistema) e diagramas comportamentais (que mostram o comportamento dinâmico e interações no sistema).

Diagrama de Classes

Representa a estrutura estática do sistema, mostrando classes, atributos, métodos e os relacionamentos entre elas.

  • Uso principal: Modelar a estrutura estática de um sistema orientado a objetos
  • Quando usar: Durante o design detalhado do sistema
  • Componentes: Classes, interfaces, atributos, métodos, relacionamentos
Criar no draw.io

Diagrama de Objetos

É uma instância do diagrama de classes em um dado momento, mostrando objetos concretos e seus relacionamentos.

  • Uso principal: Mostrar "fotografias" do sistema em um momento específico
  • Quando usar: Para explicar estruturas complexas com exemplos concretos
  • Componentes: Objetos (instâncias de classes), links (instâncias de associações)
Ver exemplo

Diagrama de Pacotes

Organiza o modelo em agrupamentos lógicos (pacotes), mostrando a decomposição do sistema em subsistemas ou módulos.

  • Uso principal: Organizar elementos do modelo em grupos lógicos
  • Quando usar: Para estruturar sistemas grandes e complexos
  • Componentes: Pacotes, dependências entre pacotes
Aprender mais

Diagrama de Componentes

Mostra os componentes de alto nível do sistema e suas interdependências, útil para visualizar a arquitetura de sistema.

  • Uso principal: Modelar componentes de software e suas relações
  • Quando usar: Durante o design de arquitetura
  • Componentes: Componentes, interfaces, dependências, portas
Ver exemplos

Diagrama de Casos de Uso

Descreve as funcionalidades do sistema do ponto de vista do usuário (atores externos), fornecendo uma visão de alto nível.

  • Uso principal: Capturar requisitos funcionais do sistema
  • Quando usar: No início do processo de desenvolvimento
  • Componentes: Atores, casos de uso, relacionamentos
Guia completo

Diagrama de Sequência

Detalha interações entre objetos ao longo do tempo, enfatizando a ordem das mensagens trocadas.

  • Uso principal: Modelar interações entre objetos em um cenário
  • Quando usar: Para detalhar como objetos colaboram em um cenário específico
  • Componentes: Lifelines, mensagens, fragmentos combinados

Diagrama de Atividades

Representa fluxos de trabalho ou processos (sequência de atividades e decisões), semelhante a um fluxograma.

  • Uso principal: Modelar processos de negócio e fluxos de trabalho
  • Quando usar: Para visualizar algoritmos ou processos complexos
  • Componentes: Ações, decisões, ramificações, fusões, junções
Tutorial prático

Diagrama de Máquina de Estados

Descreve os possíveis estados de um objeto e as transições (mudanças de estado) causadas por eventos.

  • Uso principal: Modelar o comportamento baseado em estados de um objeto
  • Quando usar: Para objetos com comportamento complexo baseado em estados
  • Componentes: Estados, transições, eventos, ações

Dica de uso prático

Embora existam muitos tipos de diagramas, na prática nem todos são usados em um mesmo projeto. Os diagramas mais comuns em projetos de software típicos incluem casos de uso, classes, sequência e atividades. A escolha dos diagramas depende do contexto e dos aspectos do sistema que se deseja enfatizar ou comunicar.

Relacionamentos e Interações

Os relacionamentos definem como os elementos do modelo se conectam ou se influenciam. Compreender os diferentes tipos de relacionamentos é fundamental para criar modelos UML precisos e significativos.

Relacionamentos em Diagramas de Classes

Associação

Relação estrutural que conecta duas classes, indicando que objetos de uma podem se relacionar com objetos da outra.

Exemplo: Aluno está associado a Curso (um aluno se matricula em cursos).

  • Representação: Linha simples entre classes
  • Multiplicidade: Indica quantas instâncias participam (1, 0..1, *, 1..*)
  • Navegabilidade: Pode ser unidirecional ou bidirecional

Agregação

Um tipo especial de associação que indica uma relação todo-parte ("é parte de"), porém com baixo acoplamento. A classe agregada (parte) pode existir independentemente do todo.

Exemplo: uma Universidade agrega vários Departamentos; se a universidade for fechada, os departamentos poderiam, em teoria, existir por si sós.

  • Representação: Losango vazio no lado do todo
  • Característica-chave: Partes podem existir sem o todo
  • Ciclo de vida: Independente entre todo e parte

Composição

É uma forma mais forte de agregação (também chamada de relação todo-parte compósita). Indica propriedade exclusiva e dependência de vida: a parte não existe sem o todo.

Exemplo: um Pedido compõe Itens do Pedido; se o pedido é deletado, os itens também são.

  • Representação: Losango preenchido no lado do todo
  • Característica-chave: Partes não podem existir sem o todo
  • Ciclo de vida: As partes são criadas e destruídas com o todo

Generalização (Herança)

Relacionamento em que uma classe especializada (subclasse) herda atributos e operações de outra classe mais geral (superclasse).

Exemplo: Aluno é uma especialização de Pessoa.

  • Representação: Seta com linha sólida e triângulo vazado apontando para a superclasse
  • Relacionamento: "é-um"
  • Implementação: Herança em linguagens orientadas a objetos

Relacionamentos em Diagramas de Casos de Uso

Include (Inclusão)

Um caso de uso base incorpora explicitamente o comportamento de outro caso de uso em seu fluxo.

Exemplo: "Comprar Produto" inclui "Processar Pagamento".

  • Representação: Seta tracejada com «include»
  • Obrigatoriedade: Sempre ocorre quando o caso de uso base é executado
  • Propósito: Reutilizar funcionalidades comuns

Extend (Extensão)

Um caso de uso opcional que estende o comportamento de um caso de uso base em certos pontos de extensão.

Exemplo: "Oferecer Desconto" estende "Finalizar Compra" sob certas condições.

  • Representação: Seta tracejada com «extend»
  • Obrigatoriedade: Ocorre apenas se condições específicas forem atendidas
  • Pontos de extensão: Locais específicos onde o comportamento pode ser estendido

Interações em Diagramas Dinâmicos

Nas interações dinâmicas modeladas em diagramas de sequência ou comunicação, o foco está em como os objetos se relacionam ao trocar mensagens. Essas mensagens podem representar chamadas de métodos, envio de sinais ou criação/remoção de objetos.

Diagrama de Sequência Diagrama de Comunicação
Organiza interações linearmente no tempo (de cima para baixo) Organiza interações em uma rede de objetos
Enfatiza a ordem temporal das mensagens Enfatiza as relações estruturais entre objetos
Mostra a duração de operações com barras de ativação Numera mensagens para indicar a ordem
Melhor para visualizar o fluxo sequencial Melhor para visualizar a rede de colaboração

Em ambos os casos, as interações ilustram colaborações entre partes do sistema para realizar um determinado cenário ou caso de uso. Esses diagramas ajudam a responder perguntas como "quem faz o quê?" e "em que ordem?". Eles complementam os diagramas estáticos, fornecendo a visão de como o comportamento flui através das estruturas definidas.

Modelagem Avançada

A UML oferece mecanismos para adaptar e aprofundar a modelagem conforme as necessidades do sistema. Estes conceitos avançados permitem maior expressividade e precisão na modelagem.

Estereótipos

Os estereótipos permitem estender a linguagem adicionando classificações específicas aos elementos. Por exemplo, pode-se marcar uma classe com o estereótipo «interface» para indicar que se trata de uma interface.

«boundary»
class TelaLogin { ... }

«controller»
class ControladorAcesso { ... }

«entity»
class Usuario { ... }

Estereótipos comuns incluem:

  • «boundary» - Classes de interface com o usuário
  • «control» - Classes controladoras
  • «entity» - Classes de dados/entidades
  • «service» - Classes de serviço
  • «repository» - Classes de acesso a dados

Perfis UML

Um Perfil UML é um conjunto de estereótipos, valores taggeados e restrições personalizado para um domínio ou propósito específico. Perfis UML permitem adaptar a UML para áreas como sistemas embarcados, modelagem de negócios, redes, entre outros, sem alterar a linguagem base.

Exemplos de Perfis UML

  • SysML - Para engenharia de sistemas
  • MARTE - Para sistemas de tempo real e embarcados
  • SoaML - Para arquiteturas orientadas a serviços
  • UML Testing Profile - Para modelagem de testes
Ver especificação

OCL (Object Constraint Language)

OCL é uma linguagem de especificação textual para definir restrições e expressar lógica que não é facilmente visualizada em diagramas. Com OCL, pode-se adicionar invariantes, pré-condições e pós-condições de operações.

context Conta inv saldoPositivo:
   self.saldo >= 0

context Conta::sacar(valor : Real) : Boolean
pre: valor > 0 and valor <= self.saldo
post: self.saldo = self.saldo@pre - valor

OCL enriquece o modelo com detalhes precisos de regras de negócio ou condições, garantindo uma especificação mais completa do sistema além do que os diagramas por si só oferecem.

Agregação vs Composição

Agregação: Relação todo-parte na qual a parte pode existir independentemente do todo. É indicada por um losango vazio no lado do todo no diagrama de classes.

Composição: Relação todo-parte forte em que a parte não pode existir sem o todo. Indica propriedade e responsabilidade de vida: quando o todo é destruído, as partes também são. Representada por um losango preenchido no lado do todo.
